# CLAUDE.md

Guidance for Claude Code (and future contributors) working in this repo.

## Project

EMS field documentation PWA. React + TypeScript + Vite, offline-first via Dexie (IndexedDB). No backend — all data is local to the device. See README.md for the user-facing feature list and setup instructions.

## Architecture

- `src/db.ts` — Dexie schema (`CallRecord`, `Shift`, `Settings`, `Hospital`, `Medication`, `InterventionDef`) and CSV export. This is the source of truth for what a saved call looks like. Schema is versioned (`this.version(N).stores(...)`) — adding a table or index means a new version block, not editing an existing one.
- `src/app/callForm.ts` — `CallForm` type (the in-progress/editable shape) and conversions: `blankForm()` (new call), `callToForm()` (load a `CallRecord` for editing). Adding a field to a call means touching both this file and `db.ts`.
- `src/app/App.tsx` — top-level state, screen routing, `saveCall()` (the only place `CallForm` is written back into a `CallRecord`).
- `src/app/screens/` — top-level screens (`HomeScreen`, `NewCallScreen`, `StatsScreen`, `MapScreen`, `SettingsScreen`, `ExportScreen`). `MapScreen` is a primary bottom-nav tab; `ExportScreen` is not — it's reached via an "Export Data" button inside Settings (see Decisions Log).
- `src/app/screens/sections/` — the card sections composed inside `NewCallScreen` (vitals, interventions, IV access, transport, acuity, notes, etc.). Each section takes `{ f: CallForm, setFld: SetFld, c: ThemeColors }` and is self-contained.
- `src/app/components/` — shared UI primitives (`FormCard`, `CardHead`, modals, chart components).
- `src/app/callStats.ts` / `shiftStats.ts` — derived stats for the Home and Stats screens.
- **Shifts** are manually logged, not live-tracked: `src/app/shiftForm.ts` (`ShiftDraft`, `blankShiftDraft()`, datetime-local conversions), `src/app/components/ShiftManagerModal.tsx` (the Add/History bottom sheet — drag-to-dismiss via the handle, both tabs grid-stacked to the same height so switching tabs doesn't resize the sheet), `src/app/components/ShiftPill.tsx` (single header pill showing unit + live elapsed time if a shift is open). A call's `date`/`unitNum`/`unitType` are always derived from its tagged shift (`f.shiftId` → `shifts.find(...)`), never stored independently on the form.
- **Hospitals, Medications, and (non-fixed) Interventions are user-editable lists, not constants.** All three live in dedicated Dexie tables (`db.hospitals`, `db.medications`, `db.interventions`), seeded once on first load from `HOSPITALS`/`DEFAULT_MEDS`/`DEFAULT_INTERVENTIONS` in `constants.ts`, then managed entirely from `SettingsScreen.tsx` (add/delete, CRUD flows mirrored across all three). `App.tsx` holds them as top-level state and passes the *live* array down into `NewCallScreen` every render — a `CallForm` never freezes a copy of these lists, so editing an older call always shows whatever hospitals/medications/interventions currently exist, not what existed when that call was first saved.
- **Oxygen and Medication are NOT part of the `interventions` table** — they're fixed, always-present rows hardcoded directly in `InterventionsCard.tsx` with their own dedicated sub-UI (O2 type/liters; saline/LR/medication list), because their complexity doesn't fit the generic toggle+note model. `SettingsScreen.tsx`'s "Manage Interventions" shows them as static, non-deletable `LockedInterventionRow`s purely for user communication ("these are always included") — deleting them isn't possible because they were never DB rows to begin with, so there's nothing to delete.
- **`InterventionDef` (`db.interventions`) is scoped per mode** (`"trauma" | "medical"`), with its own `order` field for manual reordering (swap-with-adjacent-sibling, no drag-and-drop lib) and a per-definition `notesEnabled` flag that adds a notes textarea to that intervention's row in the call form when toggled on. `CallForm`/`CallRecord.interventions` is `{ name: string; note?: string }[]` — a flat list of whichever definitions are currently toggled on for that call, replacing the old fixed `tCspine`/`tBackboard`/`tSplint`/`tBandage`/`leadOn`/`leadInterp` fields (kept on `CallRecord` as optional/legacy, unwritten-but-readable, exactly like the `zofran`/`toradol` precedent below). `interventionsSummary()` in `db.ts` renders either the new field or falls back to the legacy booleans for old records, feeding a single "Interventions" CSV column and the PDF export.
- **Map tab (`MapScreen.tsx`) — pins on a Leaflet/OpenStreetMap map, entirely independent of Calls/Shifts.** `db.locations` (a saved pin: name, category, lat/lng, optional `address`/`note`/`order`) and `db.locationCategories` (user-editable, like Hospitals — seeded from `DEFAULT_LOCATION_CATEGORIES` in `constants.ts`, each with an auto-assigned or manually-overridden `color` and a per-category `orderLocked` flag) are separate Dexie tables, unrelated to `CallRecord`/`Shift`. Map tiles and Nominatim (search/reverse-geocode) calls require a live connection — the one part of this app that isn't fully offline — but all pin data itself is local, same as everything else. A pin's `order` is scoped per category (swap-with-adjacent-sibling, same pattern as `InterventionDef.order`) and drives both List view's row order and the numbered badge shown inside each pin's map marker; reordering is only exposed while a single category (not "All") is selected, and can be locked per category via `LocationCategory.orderLocked` to prevent further changes. A preset "Station" pin/category (gray, fixed address) is seeded once alongside the default categories — see `STATION_LOCATION`/`STATION_CATEGORY` in `constants.ts`.
- **Export is not a bottom-nav tab.** It's reached via an "Export Data" button inside Settings (`SettingsScreen.tsx`) or a shortcut link on the Home dashboard; `App.tsx`'s `exportReturnScreen` state remembers which one opened it so `ExportScreen`'s back button returns to the right place. Its bottom-nav row still renders (for jumping straight to Map/Home/Stats/Settings from there), just with no tab of its own highlighted.
- **`src/app/geocoding.ts`** holds the shared Nominatim helpers (`reverseGeocode`, `forwardGeocode`, `formatAddress`, `fetchWithTimeout`) used by both `MapScreen.tsx` (pin add/search) and `App.tsx` (Hospitals auto-sync below) — don't re-duplicate these inline in a screen file. Every geocoding call goes through `fetchWithTimeout` (an 8s `AbortController` timeout) since a bare `fetch` on a dead connection can hang far longer than a user would wait; a timed-out/failed lookup is always treated as "couldn't find it," never surfaced as a hard error. `forwardGeocode` hard-bounds its search box (`bounded=1`) around a bias center rather than just soft-biasing it — confirmed against the real API that a soft bias can silently resolve a bare name to a same-named place on the other side of the world (e.g. "Southern Regional hospital" → a hospital in Belize) when there's no strong local match; hard-bounding forces a clean "no match" instead of a wrong-country pin.
- **The "Hospitals" map category (`HOSPITAL_CATEGORY` in `constants.ts`) is a locked category kept in sync with `db.hospitals`.** An `App.tsx` effect watches the `hospitals` list and, for each hospital without a same-named pin yet, calls `forwardGeocode("<name> hospital", ...)` and adds one automatically — silently, no popup. A name that can't be confidently (and correctly) geocoded alone (e.g. "St. Joe's", "CHOA Scottish Rite") stays unresolved: a red "⚠ No map location" warning + "Add Location" button appears on that hospital's own row in Manage Hospitals (`SettingsScreen.tsx`, the only place hospitals are ever added), and only opens the manual-address prompt (`activeHospitalPrompt` in `App.tsx`) when the user taps it — never automatically. Every geocode result (auto or manual) is validated with `isGeorgiaResult()` — hospitals in this app are expected to be in Georgia, so a match outside GA is treated exactly like no match at all, surfacing the same warning/prompt instead of silently placing a wrong pin. `LocationCategory.locked` (distinct from `orderLocked`) hides that category's delete button in Settings — `requestDeleteLocationCategory` also refuses server-side, not just in the UI. Deleting a hospital from the Hospitals preset list never removes its pin, same rule as every other preset in this app.

### Deleting a hospital or medication

Both go through a shared pattern in `App.tsx`: `request…()` sets a `delete…Target` id, `confirm…()` performs the actual `db.…delete()`. Before confirming, a usage count is computed against `allCalls` (`c.hospital === name` / `c.meds?.some(m => m.name === name)`) and, if non-zero, swapped into `DeleteModal`'s `message` prop as an explicit warning naming how many saved calls reference it — this is intentional, not incidental: deleting from the list must never silently drop context about calls that already used it. Deleting never mutates historical `CallRecord` data (`hospital` is a free string, `meds` entries are copied by value at save time), it only removes the option from future calls.

### Adding a new field to the call form

1. Add it to `CallRecord` in `src/db.ts`.
2. Add it to `blankForm()` and `callToForm()` in `src/app/callForm.ts`.
3. Wire it into the `record` object in `saveCall()` in `src/app/App.tsx`.
4. Build a section component in `src/app/screens/sections/` and render it in `NewCallScreen.tsx`.
5. Include it in CSV/PDF export if it should be reportable (`src/db.ts` `callsToCSV`, `src/app/screens/ExportScreen.tsx`).

Styling is inline (no CSS framework beyond a Tailwind reset) — match the existing pattern in sibling section files (tab toggles, `FormCard`/`CardHead` wrappers) rather than introducing new patterns.

## Decisions Log

Record durable design preferences and conventions here — not a changelog of every feature shipped (git history covers that). An entry belongs here only if it's a **core design preference or recurring convention**, especially one the user explicitly corrected or pushed back on. Routine "added field X" / "built feature Y" notes do not belong here once the feature is merged — keep only the underlying preference the work revealed, not the work itself.

- **Confirmation/warning dialog copy should be short and plain — one sentence, no hedging.** The user tightened the "Clear All Data" warning through several rounds of feedback, converging on "Deletes all calls and shifts, and deletes all presets to their default settings." Default to this terser register for new confirmation copy rather than a longer explanatory paragraph.
- **"Presets" is the user's preferred umbrella term for Hospitals, Medications, Interventions, and Chief Complaints.** Use "presets" in UI copy and docs instead of naming all four every time — the user corrected more specific phrasing back to this term.
- **Deleting a preset item (hospital/medication/intervention/complaint) must never mutate historical `CallRecord` data, and a non-zero usage count must always be surfaced as an explicit warning before confirming** — never a silent delete. This is a firm, repeated architectural preference, not a one-off.
- **When a fixed field is replaced by a user-editable one, the old field stays on `CallRecord` as optional/legacy (unwritten but readable)** purely for CSV/PDF fallback on pre-migration records. Never delete historical export data for the sake of a cleaner schema.
- **"Clear All Data" must reset everything user-editable (calls, shifts, and all presets) back to a fresh-install state, reseeded immediately** — not just transactional data, and not left empty until next load.
- **Calls and Shifts must export/import as one combined feature, never two separate ones.** The user explicitly corrected an initial design that gave Shifts their own standalone Export/Import — a call is fundamentally linked to its shift (`shiftId`, with `unitNum`/`unitType`/`date` always derived from the tagged shift), so splitting them apart would re-orphan every imported call. `recordsToCSV`/`csvToRecords` in `src/db.ts` round-trip both through one CSV, joined by a `ShiftStart` key. Treat any future data type this tightly coupled to an existing one the same way.
- **UI conventions worth preserving:** unit labels render without a dash (`B12` not `B-12`); viewport pinch-zoom is locked to stop iOS auto-zoom on the app's compact inputs; sliders and similar controls should be tap-to-set, not drag-only; modals with tabs keep every tab mounted in a grid stack so switching tabs never resizes the sheet.
- **`PhoneShell.tsx`'s two wrapper divs must use `height: 100dvh`, never `minHeight`.** With `minHeight`, a screen's `flex: 1; overflow-y: auto` content area never gets a bounded height to scroll within — the whole shell just grows to fit content instead, and that overflow either scrolls the whole page (which is what broke the fixed `BottomNav`, see next bullet) or, once the page itself is locked from scrolling, just gets silently clipped. `height` caps the shell so inner scroll areas actually scroll.
- **`html`/`body` must never scroll — only the app's own `overflow-y: auto` content divs should** (`theme.css`: `html, body { height: 100%; overflow: hidden }`). If the page itself can scroll (even a stray pixel from overscroll bounce), mobile Safari's dynamic toolbar hides/shows mid-scroll and the fixed `BottomNav` can visually glitch or disappear during that transition. This depends on the `height: 100dvh` fix above — the two only work correctly together.
- **A screen's map/search/status overlays need their own explicit `z-index` stacking context** (a positioned ancestor div with a numeric `z-index`, not just `position: relative`) **when third-party UI (Leaflet) is layered underneath.** Leaflet's own controls carry z-index up to 1000; without a containing stacking context they can escape into the page's root context and cover unrelated fixed elements like `BottomNav`.

## Workflow notes

- Run `npx tsc --noEmit` before considering a change done — this project has no test suite, so the type checker is the main safety net.
- `npm run build` (vite build) should also succeed before shipping a change.
- Do not run `git commit` (or push) in this repo — the user commits their own changes. Leave changes staged/unstaged in the working tree and let the user review and commit them.
